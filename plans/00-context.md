# Shared Context — read this before any plan in this folder

## Project

ESP32 fleet-status dashboard for Clausen Engineering. Internal ops tool: shows which devices run which firmware/hardware, which need updates, which can be updated remotely (OTA) vs need a physical visit. Static site on GitHub Pages. Screenshots are used in a personal portfolio, so static visual quality matters; runtime animations matter less.

## Architecture (do not change)

- **Single-file frontend:** everything (HTML/CSS/JS) lives in `index.html` (~4,400 lines). No framework, no bundler, no build step. Keep it that way.
- **Data pipeline:** `scripts/build_dashboard_data.py` (Python 3.12, stdlib only). Fetches per-device data from the monitoring API, writes `data/dashboard-data.json`. Triggered by GitHub Actions (`.github/workflows/update-dashboard-data.yml`, every 30 min + manual) and by the user's device-deployment script.
- **CDN deps:** Bootstrap 5.2.3, Bootstrap Icons, Chart.js, Google Fonts (Fraunces, Inter, JetBrains Mono).
- **Theme:** dark warm brown (`#1B1713`) with cream text, gold/red/teal accents, CSS custom properties near top of `index.html`. Keep the existing look — extend it, don't redesign.

## Monitoring API

Base `https://monitoring-api.3aentreprise.com`, HTTP Basic Auth via env vars `API_USERNAME` / `API_PASSWORD`. Endpoints currently used by the pipeline:

- `GET /devices/{macAddress}` — device name, settings (sensors array, sensorLocation), location
- `GET /firmwares/latest?deviceMac={macAddress}` — `version`, `buildDate`

The firmware (separate repo `../sensorhub-data-collector`) also POSTs logs/events/errors to this API and reports health telemetry, but this dashboard deliberately stays version/deployment-focused — device health lives on the 3A Console, not here.

## Key data files

- `data/devices.json` — hand-edited registry. Structure: `sections[] → { id, label, description, api_base, tracks[], devices[] }`. Each device: `{ mac, label, track }`, optionally `hardware`, `components`. 16 sensor-hub devices + 2 controllers.
- `data/version-changes.json` — hand-edited release history: `{ latest_version, repo_name, last_commit_date, version_changes: [{ version, date, components[], description }] }`. Component tags: `core`, `OneWire`, `MODBUS`, `digital`, `switch`, `virtual`, `algae`.
- `data/dashboard-data.json` — generated by pipeline, committed by CI (verify; CLAUDE.md claims it's gitignored but recent commits say "Update dashboard data"). Never hand-edit.

## Frontend structure (index.html)

- All UI state in one `dashboardState` object (~line 2311).
- Entry point `initializeDashboard()` (~line 2342) — parallel fetch of the 3 JSON files.
- Key renderers: `renderDeviceCards()`, `renderDeviceTable()`, `renderVersionHistory()`, `renderStatusBreakdown()` (Chart.js doughnut), `applyFiltersAndRender()`.
- Sections: Overview (hero + 4 stat cards + repo metadata), Fleet readiness (doughnut + status breakdown), Device browser (search, status/component filter chips, card/list toggle, sort, copy-view URL), Sensor topology (location groups + component atlas), Version timeline, Device detail drawer (slide-in panel).
- Status logic (~line 3972): version vs track target → `Up to date` / `Patch available` (same major.minor, behind patch) / `Needs update` (behind major/minor) / `Not deployed` / `Unknown`.
- XSS: escape all device-sourced strings via existing `escapeHtml()`.
- Versions normalized with `v` prefix (`v3.21.0`).

## Conventions for agents

- Match existing code style in `index.html` (vanilla JS, no modules, functions in the existing script block).
- Test locally: `python -m http.server 8000` from repo root. Pipeline run needs API creds — if unavailable, hand-craft a sample `data/dashboard-data.json` extension to test frontend changes, but do not commit hand-edited dashboard-data.json.
- Commit messages: short imperative, like existing history ("Update dashboard data", "Simplify device card version details").
- Do not add npm/build tooling, frameworks, or external JS/CSS files.
