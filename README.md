# ESP32 Sensor Monitoring - Device Status Dashboard

A live fleet-operations dashboard for ESP32 sensor devices in the 3A console monitoring system. Tracks firmware versions (device-reported and registry), deployment status, OTA history, and automated stale-device alerts.

[Device status website](https://clausen-engineering.github.io/3a-console-sensor-device-status/)

## Features

- **Live Device Telemetry**: Reported firmware version and battery level from the monitoring API
- **OTA Audit Trail**: Track successful firmware updates per device with timestamps and version codes
- **Device Status**: Six-category status (Up to date / Patch available / Needs update / Not deployed / In development / Unknown)
- **Fleet Alerts**: Automated GitHub issues for behind-updates (>7d old release)
- **Deployment Registry**: Manual fallback for devices without live telemetry; optional to maintain post-flash

## Data Pipeline

The dashboard reads from `data/dashboard-data.json`, generated by `scripts/build_dashboard_data.py` every 6 hours via GitHub Actions.

### Dashboard Data Schema

#### Per-Device Fields (in `sections[].tracks[].devices[]`)

**Core identification:**
- `mac` — MAC address (lowercase, colon-separated)
- `label` — human-readable device name
- `location` — deployment location

**Live telemetry (from API):**
- `reported_version` — firmware version from device's latest log (normalized with "v" prefix, empty string if unavailable)
- `battery_level` — lastLog.batteryLevel percentage (null if absent)

**Version state:**
- `version` — **source of truth**: reported_version if reported, else registry installed_firmware_version, else track latest
- `version_source` — "reported" | "registry" | "" (indicates which source was used)
- `version_mismatch` — boolean; true when both reported and registry versions exist and differ
- `declared_deployment_version` — registry installed_firmware_version (kept for audit/mismatch display)

**OTA & update availability:**
- `ota_history` — array of `{date: "ISO 8601", version: "v3.16.6", version_code: 3160600}`, newest first, max 10 entries; sourced from API "OTA Update Completed" events
- `pending_ota_version` — next firmware available for update (from /firmwares/latest), empty string if none or device not ota_capable
- `pending_ota_source` — "api" | "registry" | "" (source of pending version)
- `target_version` — track's latest_version (for status classification)

**Hardware & capabilities:**
- `hardware` — hardware model ID (e.g., "sensor-hub-v1.5-esp32-s3-n16r8"); from API hardwareInfo.board or registry
- `ota_capable` — boolean; whether device supports OTA updates
- `hardware_eol` — boolean; whether hardware is end-of-life
- `hardware_max_firmware` — highest firmware version this hardware can run (or empty string)

**Deployment metadata:**
- `initial_deployed`, `last_deployed` — ISO dates of first and most recent deployments
- `components` — sensor/feature tags (core, OneWire, MODBUS, digital, switch, virtual, algae, etc.)

#### Section/Track Structure

Sections group device types (e.g., "sensor-hub"). Each section contains:
- `id`, `label`, `description`
- `repo_name` — source firmware repository (e.g., "glaecier-sensorhub-data-collector")
- `latest_version` — track's target firmware version
- `tracks` — list of firmware tracks with version history and device lists

## Updating Device Data

The GitHub Actions workflow in [`.github/workflows/update-dashboard-data.yml`](.github/workflows/update-dashboard-data.yml) runs every 6 hours:

1. Runs `python scripts/build_dashboard_data.py` (requires API credentials)
2. Generates `data/dashboard-data.json` with live telemetry from the monitoring API
3. Commits and pushes the updated file to GitHub Pages

### Required GitHub Secrets

- `API_USERNAME` — monitoring API username
- `API_PASSWORD` — monitoring API password
- `SENSORHUB_REPO_PAT` — GitHub token for accessing the firmware repository (optional, for reading firmware metadata)
- `GITHUB_TOKEN` — automatically available; used for fleet alerts

Auth strategy: scripts attempt Bearer JWT via `POST /auth/token` with form-encoded credentials, falling back to HTTP Basic Auth.

### Security Note

**The GitHub Pages site is public.** No credentials are sent to the browser; all API access is server-side in GitHub Actions. The dashboard consumes only committed JSON files. However, be aware that:
- Device MAC addresses and physical locations are visible in the published data
- The commit history is public

Repository visibility (public vs. private) and data exposure are open decisions for the owner to make based on your operational security requirements.

## Understanding Device Status

Status is calculated by comparing device firmware version to the track's target:

- **Up to date** — version equals target_version
- **Patch available** — same major.minor, behind on patch (e.g., v3.21.1 available, device has v3.21.0)
- **Needs update** — behind on major or minor version
- **In development** — no target_version set for the track
- **Not deployed** — device has no installed version record
- **Unknown** — version data is missing or cannot be determined

## Fleet Alerts

The workflow in [`.github/workflows/update-dashboard-data.yml`](.github/workflows/update-dashboard-data.yml) runs `scripts/check_fleet_alerts.py` after each data pipeline run, creating GitHub issues for:

1. **Behind schedule** — status "Needs update" AND the target release date is >7 days old

Issues are labeled `fleet-alert` with title format `[fleet-alert] <label> (<mac>): <rule-slug>`. When a rule stops firing, the issue is auto-closed.

## Deployment Workflow Integration

Post-flash with new firmware:

**Option 1 (recommended): Rely on device-reported version**
- Device automatically reports its version via the monitoring API
- `build_dashboard_data.py` reads it and displays it as source of truth
- No manual registry updates needed

**Option 2 (optional fallback): Update the registry**
- Edit `data/devices.json` to set `installed_firmware_version` for the device
- Edit `devices/*/version.json` in `glaecier-sensorhub-data-collector` to set `deployment_version`
- These are used only when device-reported version is unavailable
- Updating these files is no longer required if the device is reporting correctly

## Custom Development

### Modifying the Dashboard

The dashboard is built with:
- HTML5
- Bootstrap 5.2.3 for styling
- Chart.js for visualizations
- Vanilla JavaScript (no framework)

All code is in `index.html` (~7,338 lines). To customize:

1. Edit `index.html` for layout, styling, and behavior
2. Modify CSS custom properties for theme changes
3. Update JavaScript render functions for UI changes

### Running Locally

```bash
# Serve dashboard locally
python -m http.server 8000

# Run data pipeline (requires API credentials in env: API_USERNAME, API_PASSWORD)
python scripts/build_dashboard_data.py

# Run tests
python -m unittest discover -s tests -v
```

## Troubleshooting

### Dashboard Not Loading

- Verify `data/dashboard-data.json` exists and is valid JSON
- Check `data/devices.json` and `data/version-changes.json` are present
- View browser console for JavaScript errors

### Stale or Missing Device Data

- Check GitHub Actions workflow runs in the repo (Workflows → "Update Dashboard Data")
- Verify `API_USERNAME` and `API_PASSWORD` secrets are set correctly
- Check workflow logs for API errors
- If offline, the dashboard shows registry data as fallback

### Devices Showing Wrong Version

1. Check if `version_source` is "registry" (device not reporting) or "reported" (from device)
2. If mismatch detected, both versions appear in the device drawer
3. Update `data/devices.json` (registry) for non-reporting devices, or wait for the device to report
